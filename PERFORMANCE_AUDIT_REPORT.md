### Critical Issues

**1. Hero images are 2.6 MB and 3.4 MB — the single biggest performance problem on the site.**

The three hero slides (hero-1.jpg at 2.6 MB, hero-2.jpg at 3.4 MB) are the LCP candidates. At 3G speeds, a 3.4 MB image takes ~8-10 seconds just to download. Next.js Image Optimization with `formats: ['image/avif', 'image/webp']` is configured but only applies when images are served through the built-in image optimizer — this depends on the hosting platform's image optimization API being available.

**2. Same images reused across the entire site for different contexts.**
- `/images/hero-1.jpg` is used for: Hero slide, Welcome section image, About page, Admissions page, Events hero, Curriculum overview hero, past events, Shop section, and CTAs
- No dedicated photography for sections like Welcome, About, Admissions, Events — they all reuse hero images inappropriately

**3. No AVIF/WebP source files.**
- `next.config.ts` has `formats: ['image/avif', 'image/webp']` but there are no actual .avif or .webp files in `public/images/`
- Relies entirely on Next.js runtime image optimization, which may not be available depending on hosting configuration

**4. `sizes` attributes are inconsistent.**
- Hero images use `sizes="100vw"` (correct)
- Welcome uses `sizes="(max-width: 1024px) 100vw, 50vw"` (correct)
- But `srcset` is only generated by Next.js `Image` component — actual width variants depend on `deviceSizes` and `imageSizes` in config

**5. Priority/fetchpriority on hero slides is suboptimal.**
- Only the first slide gets `priority` and `fetchPriority="high"`
- But the LCP hero image may be slide 2 or 3 depending on the auto-rotation timer

### Google Fonts — Critical Issue

```css
/* globals.css:1 - THIS IS RENDER-BLOCKING */
@import url('https://fonts.googleapis.com/css2?family=Montserrat:wght@400;500;600;700;800;900&family=Open+Sans:wght@300;400;500;600;700&display=swap');
```

Using `@import` in CSS for Google Fonts blocks rendering until the font stylesheet is fetched. This delays FCP by ~200-400ms on fast connections and by seconds on slow connections. `display=swap` is present but the `@import` itself is the bottleneck.

**Fix:** Replace with `next/font`:
```tsx
// app/layout.tsx
import { Montserrat, Open_Sans } from 'next/font/google';

const montserrat = Montserrat({
  subsets: ['latin'],
  variable: '--font-heading',
  display: 'swap',
});

const openSans = Open_Sans({
  subsets: ['latin'],
  variable: '--font-body',
  display: 'swap',
});
```

Then use CSS variables `var(--font-heading)` and `var(--font-body)` — which the codebase already uses. This eliminates the render-blocking external request entirely.

### Finding: `Locations.tsx` is unnecessarily a Client Component

`app/components/Locations.tsx` uses `'use client'` but contains zero interactivity — no hooks, no state, no event handlers, no browser APIs. This forces the entire component subtree to be client-rendered, adding unnecessary JS to the bundle.

**Fix:** Remove `'use client'` from `Locations.tsx`.

### Finding: `ActivityDetailContent.tsx` is unnecessarily a Client Component

`app/co-curricular/[slug]/ActivityDetailContent.tsx` uses `'use client'` only to call `use(params)`. With Next.js 16, params can be awaited in Server Components directly. This entire component can be a Server Component, eliminating framer-motion and lucide-react from the server-rendered bundle.

**Fix:** Remove `'use client'`, convert to async Server Component.

### Data Fetching Pattern

All data fetching is synchronous from static JS modules (`app/lib/data/index.ts`). There are no `useEffect` data fetching patterns, no API calls from the client, and no external data sources. This is good — no unnecessary client-side fetching overhead.

However, this means **every page is static** with no dynamic data from a CMS or backend. Adding a headless CMS (Strapi, Sanity, Contentful) would enable dynamic content without performance cost if done with Server Component data fetching.

### LCP Root Cause Analysis

The LCP element is the hero slide image. The current flow:
1. Browser parses HTML → discovers `globals.css` with `@import url(...)` for Google Fonts → **blocks rendering**  
2. Browser starts font download (~200ms)  
3. Browser discovers Hero component → requests 2.6 MB JPEG  
4. Browser downloads 2.6 MB at 3G speed: ~6-8 seconds  
5. Image renders → LCP fires

**Total LCP estimate: ~4-6 seconds on 3G, ~2-3 seconds on fast 4G.**

### Fix Strategy
1. Replace Google Fonts `@import` with `next/font` → saves ~300ms FCP
2. Convert hero-1.jpg to hero-1.avif (~150 KB) → saves ~2-3s on LCP
3. Preload hero-1.avif in `<head>` → saves ~200ms discovery time
4. Target LCP: **<2.5s on 3G**
